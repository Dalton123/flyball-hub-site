# Manual Cache Revalidation

When you update content in Sanity but need the change to appear on the live site sooner than the default ISR window, you can manually revalidate the cache by calling the `/api/revalidate` endpoint with a correctly signed webhook payload.

## Quick Command

```bash
cd apps/web
node scripts/generate-revalidation-curl.mjs "/blog/your-post-slug"
```

This prints a ready-to-run `curl` command containing the correct `sanity-webhook-signature` header generated from your local `SANITY_REVALIDATE_SECRET`.

### Example

```bash
node scripts/generate-revalidation-curl.mjs "/blog/dog-breeds-for-flyball"
```

The output will look like:

```bash
curl -X POST https://www.flyballhub.com/api/revalidate \
  -H "Content-Type: application/json" \
  -H "sanity-webhook-signature: t=1234567890123,v1=abc..." \
  -d '{"_type":"blog","slug":{"current":"/blog/dog-breeds-for-flyball"}}'
```

Copy and execute the command. A `200 OK` with `{ "revalidated": true, ... }` means the cache was purged.

## Troubleshooting

### 401 Invalid signature

If you see `Invalid signature`, the `sanity-webhook-signature` header was computed with a secret that does **not** match the `SANITY_REVALIDATE_SECRET` configured on the deployed Vercel environment.

**Common causes:**

1. **Mismatched secrets** — Your local `.env.local` contains a different `SANITY_REVALIDATE_SECRET` than the Vercel production environment. Verify in Vercel Dashboard: Project Settings → Environment Variables.
2. **Wrong secret name** — Make sure the deployed env var is named `SANITY_REVALIDATE_SECRET` (not `NEXT_PUBLIC_SANITY_HOOK_SECRET` or similar).
3. **Signature algorithm mismatch** — Only the `t=<timestamp>,v1=<base64url_sig>` format generated by `parseBody` (from `next-sanity/webhook`) is accepted. Raw HMAC-SHA256 headers are **not** accepted.

### How to verify your local signature is valid

```bash
cd apps/web
node scripts/test-webhook-sig.mjs
```

If the script returns `SUCCESS: local signature validates with parseBody`, your local secret produces valid headers against the exact same library used by `/api/revalidate`. This means the 401 is almost certainly a secret mismatch between your local machine and Vercel.

## Signature Format Details

The endpoint uses `next-sanity/webhook` (`parseBody`). The header must match:

```
sanity-webhook-signature: t=<timestamp>,v1=<signature>
```

Where `<signature>` is a base64url-encoded (no padding) HMAC-SHA256 of the literal string `<timestamp>.<json_body>` signed with the same secret.

Never attempt to sign by hand using plain `sha256(body)`. The signature is:

1. `message = timestamp + "." + JSON.stringify(payload)`
2. `signature = base64url(HMAC-SHA256(message, secret))`

The helper scripts already do this correctly.

## Updating the Deployed Secret

If the secrets are mismatched and you need to align them:

1. Generate a new strong secret (any long random string).
2. Replace it in Vercel Dashboard → Environment Variables → `SANITY_REVALIDATE_SECRET`.
3. Replace it in your local `apps/web/.env.local`.
4. Verify with `node scripts/test-webhook-sig.mjs`.
5. Redeploy the project so the new secret is active on the server.
